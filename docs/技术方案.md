# audioBlade 技术方案

## 目标

构建一款 Windows + macOS 桌面音频路由应用，具备类似 TuneBlade 的持久化能力：无需使用 Studio One、Audition 等 DAW 软件即可完成系统音频路由与跳线。

产品应支持以下音频路由场景：

- 物理音频接口（含 USB/Thunderbolt 外置声卡）
- 系统输出与应用播放
- 应用或设备输入
- 虚拟输入与虚拟输出
- 未来可选的网络输出（如 AirPlay 类流播）

## 核心产品需求

DAW 软件只能在运行时临时充当音频跳线盘，关闭后路由即中断。audioBlade 的定位是——无需打开任何 DAW，即可拥有一个永久在线的系统级音频路由方案。

最低架构要求：

- 一个后台音频引擎进程或系统服务
- 一个或多个向操作系统暴露的虚拟音频设备
- 一个编辑路由状态的 GUI
- 登录时/设备重连后可恢复的已保存路由配置

Python 应用本身无法向操作系统暴露稳定的系统音频设备。Python 可以控制 UI、配置或编排，但实时音频路径应该用原生代码实现。Windows 虚拟音频设备是驱动级别组件，Microsoft 的 SYSVAD 示例是自研虚拟音频驱动的参考路径。[2] macOS 虚拟音频设备同样需要 Core Audio 或 AudioDriverKit 级别的组件，而非普通的应用层代码。[3][4]

## 开源驱动依赖策略

MVP 阶段使用的虚拟音频驱动必须开源且兼容未来商业使用。商业发行版优先选择宽松或弱 Copyleft 许可证。

默认策略：

- 优先：MIT、BSD、Apache-2.0、MS-PL、MPL-2.0
- 审慎接受：LGPL、文件级 Copyleft（如果动态链接或文件级源码义务可管理）
- 避免作为商业版默认依赖：GPL-2.0/GPL-3.0 驱动（除非 audioBlade 本身兼容 GPL，或驱动作者单独授权商业许可）
- 避免作为默认依赖：闭源的免费软件、捐赠软件和专有虚拟线缆驱动

此为工程兼容性筛选，非法律建议。在商业安装包中捆绑驱动前，须进行正式许可证审查。

## 驱动能力对比

许可证兼容的开源候选驱动可以覆盖最底层：暴露虚拟音频端点。它们不能完整覆盖：

- 完善的路由矩阵
- 每路增益/静音
- 设备重连恢复
- 配置管理
- ASIO 级别的 Windows 工作流
- 多客户端 DAW 行为
- 终端用户安装包质量
- 采样率不匹配和时钟漂移的诊断

因此，产品策略应为：

1. 仅将许可证兼容的开源驱动作为 MVP 虚拟端点依赖
2. 在其之上构建 audioBlade 自己的路由引擎和 UI
3. 若端点可靠性或商业打包成为障碍，再规划自研驱动
4. 以 VoiceMeeter（Windows）和 BlackHole/Loopback 类工作流（macOS）为基准对标，同时避免不兼容的依赖

## 推荐的起始驱动

每个平台先选一个驱动开始。不要一开始就支持所有候选，因为早期工作应优先验证 audioBlade 的路由图、配置持久化和设备重连行为。

### Windows 起始驱动

使用 `VirtualDrivers/Virtual-Audio-Driver`。[9][10]

理由：

- 已支持 Windows 10/11
- 同时暴露虚拟扬声器和虚拟麦克风，符合初始路由用例
- 基于 Microsoft SYSVAD 方向，未来迁移到自研 SYSVAD 派生驱动在概念上一致
- 仓库声明原创代码为 MIT，Microsoft 示例派生部分为 MS-PL，符合商业使用策略
- 相比从原始 SYSVAD 起步，更接近可用的终端用户虚拟端点

除非第一里程碑明确是驱动研发，否则不要从原始 SYSVAD 开始。SYSVAD 是未来自研驱动的正确参考，但会拖慢产品体验的验证。[2]

### macOS 起始驱动

使用 `roc-streaming/roc-vad` 作为研究/MVP 候选。[13]

理由：

- MPL-2.0，比 GPL 驱动更易于在专有商业应用中使用（需正式法律审查）
- 是现代的 macOS 虚拟音频设备项目，非仅兼容旧版
- 支持创建和控制虚拟设备并暴露程序化控制接口，适合 audioBlade 管理的路由层
- 提供了一个实际的路径来验证安装、设备持久化和引擎集成，而不依赖 BlackHole 的 GPL-3.0 许可证

`Soundflower` 仅作为备选兼容性验证。[14] 其 MIT 许可证有吸引力，但该项目较旧，不应定义主要的 macOS 路线。

### 第一阶段 MVP 顺序

按此顺序构建和测试：

1. Windows + VirtualDrivers/Virtual-Audio-Driver
2. audioBlade 路由引擎：一路虚拟输入到一路物理输出
3. 配置持久化与自动恢复
4. 设备重连处理
5. macOS + Roc VAD 验证
6. 对比 VoiceMeeter/BlackHole 工作流找出功能差距

此顺序优先走最直接的 Windows 路径，避免在产品模型被验证前过早投入 macOS 驱动打包时间。

### Windows

与 VB-CABLE 对比：

- 差距：中等
- VB-CABLE 主要是一条虚拟线缆，发送到线缆输入的音频被转发到线缆输出[12]
- VirtualDrivers/Virtual-Audio-Driver 提供相同类别的能力：Windows 虚拟扬声器和虚拟麦克风，对应用可见[9]
- 实际差距可能在于安装包完善度、成熟度、支持历史和边缘情况稳定性，而非基本概念

与 VoiceMeeter 对比：

- 差距：很大
- VoiceMeeter 不仅是虚拟驱动，它是一个完整的虚拟调音台，包含硬件输入、软件输入、多总线路由、效果器、宏控制、虚拟 ASIO、多通道支持和网络流播功能[17][18]
- 开源驱动候选无法替代这一产品级调音台。audioBlade 需要自己实现路由、混音、增益、静音、配置持久化和诊断
- 对于第一里程碑，这个差距是可接受的，因为 MVP 需要虚拟端点加上基本的立体声路由器。要达到 VoiceMeeter Banana/Potato 的生产力替代品，差距则很大

Windows 建议：

- 只将 VirtualDrivers/Virtual-Audio-Driver 或 SYSVAD 派生驱动作为虚拟端点层
- 不要期望驱动依赖提供完整产品体验
- 将 VoiceMeeter 作为功能基准，而非架构依赖

### macOS

与 BlackHole 对比：

- 差距：中到大
- BlackHole 是一款现代的 macOS 虚拟环回驱动，明确面向应用到应用的音频透传，零额外延迟，支持多通道配置[15]
- Roc VAD 是现代的且为 MPL-2.0，但其主要关注点是通过 Roc Toolkit 兼容的远程对等端进行虚拟设备流播，而非通用本地环回线缆[13]
- Soundflower 是 MIT 许可证且在历史上很有用，但较旧，不应作为当前 macOS 版本的默认选择[14]

macOS 建议：

- 使用 Roc VAD 作为第一个现代许可证兼容的研究候选
- 仅将 Soundflower 用于历史对比或兼容性测试
- 将 BlackHole 作为功能基准。不要将其捆绑到专有商业构建中，除非 audioBlade 兼容 GPL-3.0 或从作者处获得单独商业许可[15]

### 对产品的影响

开源候选可以覆盖最底层：暴露虚拟音频端点。它们不能完整覆盖：

- 完善的路由矩阵
- 每路增益/静音
- 设备重连恢复
- 配置管理
- ASIO 级别的 Windows 工作流
- 多客户端 DAW 行为
- 终端用户安装包质量
- 采样率不匹配和时钟漂移的诊断

因此，产品策略应为：

1. 仅将许可证兼容的开源驱动作为 MVP 虚拟端点依赖
2. 在其之上构建 audioBlade 自己的路由引擎和 UI
3. 若端点可靠性或商业打包成为障碍，再规划自研驱动
4. 以 VoiceMeeter（Windows）和 BlackHole/Loopback 类工作流（macOS）为基准对标，同时避免不兼容的依赖

## 自研虚拟驱动可行性

自研虚拟音频驱动是可行的，但不应作为第一里程碑。工程难度不仅在于音频 DSP，主要成本在于操作系统集成、驱动生命周期、签名/公证、安装包、兼容性测试以及跨系统版本维护。

### Windows 可行性

技术路径：

- 以 Microsoft SYSVAD 示例为虚拟音频端点驱动的起点[2]
- 定制端点拓扑、通道布局、名称、格式以及到 audioBlade 引擎的路由桥接
- 构建签名驱动包和安装程序[6][11]
- 在 Windows 10 和 Windows 11 上测试常见音频 API：WASAPI 共享模式、WASAPI 独占模式、DirectSound 兼容性以及 DAW 行为

预估工作量：

- 研究验证：2-4 周
- 内部原型驱动：1-2 个月
- 可用 MVP 驱动（含安装包）：2-4 个月
- 商业级驱动（含签名、更新处理、崩溃诊断、广泛的硬件/应用兼容性）：4-9 个月以上

主要风险：

- 驱动签名与分发流程
- 内核态代码纠错不当可能导致蓝屏/内核稳定性风险
- 跨 DAW、会议应用、浏览器和游戏的应用兼容性边缘情况
- 多客户端行为与格式协商
- 调试成本远高于普通应用调试

Windows 结论：

- 可行，但成本高
- 从 SYSVAD 开始是现实的
- 对独立开发者或极小型团队，预计需要数月才能稳定到付费用户可用的程度

### macOS 可行性

技术路径：

- 使用目前 Apple 支持的音频驱动路径构建虚拟音频设备：AudioDriverKit（如适用），或 Core Audio HAL/AudioServerPlugIn 风格（尚合适时）[3][4]
- 实现虚拟输入/输出流并桥接到 audioBlade 引擎
- 处理包标识符、授权、用户批准、公证和安装流程[7]
- 在当前的 macOS 版本和 Apple Silicon/Intel（如果都需支持）上测试

预估工作量：

- 研究验证：3-6 周
- 内部原型虚拟设备：1-3 个月
- 可用 MVP 驱动/插件（含安装包）：3-5 个月
- 商业级实现（含公证、用户批准流程、设备恢复、系统更新韧性）：6-12 个月以上

主要风险：

- Apple 平台规则与驱动模型变更
- 授权与安装程序审批摩擦
- 系统更新后设备消失或权限问题
- 时钟、采样率协商与聚合/多输出行为
- 可参考的现代宽松许可证示例较少

macOS 结论：

- 可行，但产品化通常比基础原型更困难
- 许可证兼容的依赖路径有助于在投入此成本前验证产品行为

### 实际建议

除非产品需求明确为"无第三方虚拟设备依赖即可发布"，否则不要从自研驱动开始。更好的顺序是：

1. 使用许可证兼容的现有驱动构建路由引擎和 UI
2. 衡量依赖在哪些方面阻碍了商业质量：延迟、通道数、安装包、稳定性、品牌或支持
3. 仅在产品需求已得到验证后，再并行启动自研驱动验证
4. 将 Windows 和 macOS 驱动视为独立项目，而非共享跨平台模块

如果 audioBlade 包含双平台自研驱动，预估排期：

- 无自研驱动的 MVP 应用：1-3 个月
- 添加 Windows 自研驱动：额外 3-6 个月达到可用质量
- 添加 macOS 自研驱动：额外 4-8 个月达到可用质量
- 双平台商业就绪：实际视团队规模、测试深度和签名/公证摩擦程度，约 9-18 个月

## 推荐语言和技术栈

### 当前技术栈决策

推荐的长期技术栈为混合架构：

- 实时音频引擎：C++，使用 JUCE
- 控制服务：Rust
- GUI：最快 MVP 用 JUCE；引擎稳定后如需网页风格产品 UI，可用 Tauri
- Windows 后端：首选 WASAPI，可选 ASIO，后续定制驱动
- macOS 后端：Core Audio HAL / AudioServerPlugIn，后续必要时用 DriverKit
- 配置/状态：JSON 或 SQLite
- 脚本/原型：Python 仅用于工具和测试

### 为什么混合 C++ + Rust

C++ 仍然是实时音频路径最安全的选择，因为 JUCE 提供了成熟的跨平台音频设备管理、回调处理、缓冲区处理和桌面音频应用基础设施。[5]

Rust 在有价值的非实时控制层中发挥作用：

- 路由图状态
- 配置持久化
- 设备重连策略
- 进程间通信
- 日志和诊断
- 更新编排
- 配置验证
- 后台服务生命周期

关键边界是：音频缓冲区在正常运行中不得跨越 Rust/C++ 或进程边界。Rust 发送控制消息，但不处理实时音频缓冲区。

### MVP 技术栈

如果速度是优先项，先使用 C++/JUCE 单进程原型。这样可以保持初始构建小巧，避免跨语言复杂性，并快速验证音频路径。

音频路径正常工作后，如果配置管理、设备状态、诊断和更新行为的复杂度已足以证明拆分边界的合理性，再拆分出 Rust 控制服务。

### 最终产品技术栈

对于商业产品，建议：

```
C++/JUCE 音频引擎
  - 实时音频回调
  - 设备 I/O
  - 低延迟路由
  - 平台音频 API 集成

Rust 控制服务
  - 路由图所有权
  - 配置
  - 设备重连状态机
  - 诊断
  - 更新与生命周期控制

GUI
  - JUCE（最小原生应用路径）
  - Tauri（如果更丰富的 UI 迭代和 Web 风格界面变得更有价值）
```

只要实时音频路径保持在 C++ 引擎内，这种拆分不应显著降低运行时性能。主要的安装/运行时开销来自虚拟音频驱动以及（如果选择使用）Tauri 的 WebView 依赖，而非 Rust 本身。

### 为什么引擎不用 Python

Python 对于原型可以接受，但不适合本产品的核心音频路径，因为：

- 低延迟回调音频需要可预测的时序
- 向操作系统暴露音频设备需要原生驱动/插件组件
- 打包特权音频组件是平台相关的
- GIL 和依赖打包在实时路径中增加风险

Python 仍可用于 CLI 诊断、集成测试和开发工具。

## 架构

```
+--------------------+        +---------------------------+
| 桌面 GUI           | <----> | Rust 控制服务              |
| 路由编辑器          | IPC    | 配置、状态、策略            |
+--------------------+        +-------------+-------------+
                                           |
                                           | 仅控制消息
                                           v
                              +------------+-------------+
                              | C++/JUCE 音频引擎        |
                              | 实时路由图               |
                              +------------+-------------+
                                           |
                         +-----------------+-----------------+
                         |                                   |
                +--------v--------+                 +--------v--------+
                | 虚拟设备         |                 | 物理设备         |
                | 输入/输出        |                 | 外置声卡         |
                +-----------------+                 +-----------------+
```

对于第一个 MVP，Rust 控制服务可以合并到 JUCE 应用中以减少实现开销。代码中仍应保持架构边界，以便后续可以拆分出服务。

## 持久化路由模型

使用图模型而非硬编码配对。

实体：

- `AudioNode`：物理设备、虚拟设备、应用捕获流、网络接收端
- `Port`：节点上的输入或输出通道
- `Route`：从源端口到目标端口的连接
- `Profile`：命名的路由集合，含通道映射、增益和采样率策略

配置示例：

```json
{
  "name": "监听音箱",
  "routes": [
    {
      "source": "virtual:AudioBlade Input:1-2",
      "destination": "device:外置接口 Out:1-2",
      "gainDb": 0.0
    },
    {
      "source": "device:外置接口 In:1-2",
      "destination": "virtual:AudioBlade Mic:1-2",
      "gainDb": 0.0
    }
  ]
}
```

## Windows 实现

### 阶段 1：用户态 MVP

使用现有虚拟音频线缆软件进行初步验证：

- Windows 默认候选：VirtualDrivers/Virtual-Audio-Driver。它为 Windows 10/11 暴露虚拟扬声器和虚拟麦克风，原创代码标记为 MIT，包含 Microsoft SYSVAD 衍生第三方代码（MS-PL 声明）。[9][10]
- Windows 备选/参考候选：直接使用 Microsoft 的 SYSVAD 示例。MS-PL 许可证，适合作为自研驱动的基础，但它只是一个示例而非产品就绪的虚拟线缆。[2][11]
- 不作为默认：VB-CABLE 和 VoiceMeeter 类驱动，因为它们不是本项目默认商业路径的合适开源依赖。[12]
- audioBlade 控制这些设备与物理接口之间的路由。

先使用 WASAPI 共享模式。WASAPI 支持设备级别的捕获/渲染流，Microsoft 提供了端点环回录音的文档。[1]

- 枚举渲染/捕获设备
- 从虚拟输入或物理输入捕获
- 渲染到物理输出或虚拟输出
- 持久化配置并在登录时自动启动引擎

这样可以在不编写驱动的情况下获得可用的路由管理器。

### 阶段 2：更好的延迟和专业音频支持

增加：

- 安全场景下的 WASAPI 独占模式
- 面向专业外置接口的 ASIO 支持
- 源和目标采样率不同时的采样率转换
- 独立硬件时钟之间的漂移校正

### 阶段 3：自研 Windows 虚拟音频驱动

选项：

- 基于 Microsoft SYSVAD 示例构建定制虚拟音频驱动[2]
- 需要驱动签名和安装程序流程[6]
- 这预计是 Windows 端成本最高的任务

仅在 audioBlade 必须自带自研虚拟设备而非依赖第三方虚拟线缆时才需要。

## macOS 实现

### 阶段 1：使用现有虚拟设备

使用许可证兼容未来商业用途的开源虚拟音频驱动。

默认候选：

- Roc VAD：macOS 虚拟音频设备，MPL-2.0。专为实时流播到虚拟设备及从虚拟设备流播而设计，支持 macOS 10.15+，明确记录了打包/分叉要求（如更改包标识符）。[13]
- Soundflower：MIT 许可证，在 macOS 音频环回方面历史上很有用，但属于较旧系统扩展，应作为兼容性验证而非现代默认选择。[14]

不作为默认：

- BlackHole：技术性强力候选，但为 GPL-3.0。其 README 声明：应用为 GPL-3.0 时允许集成；否则需联系作者。因此不适合作为专有商业构建的默认依赖。[15]
- Background Music：GPL-2.0，对 macOS 音频路由概念是有用的参考，但不是默认的专有依赖。[16]

audioBlade 提供：

- 路由持久化
- 配置切换
- 设备重连处理
- GUI 路由矩阵

### 阶段 2：原生 Core Audio 虚拟设备

根据目标 macOS 版本，构建 Core Audio HAL AudioServerPlugIn 风格虚拟设备，或评估当前的 DriverKit 要求。[3][4]

macOS 注意事项：

- 虚拟音频设备不仅是普通应用代码
- 安装可能需要用户批准
- 授权、公证和打包值得关注[7]
- 多输出路由需要时钟漂移处理

## 网络/TuneBlade 类流播

如果 TuneBlade 的同等功能特指将 Windows 音频流播到 AirPlay 接收端，则将其视为一种可选的接收端类型。TuneBlade 本身的定位是 Windows 端将音频流播到 AirPlay 接收端的软件，因此这与 Studio One 风格的本地跳线盘用例相关但有所区别。[8]

```
音源 → audioBlade 引擎 → 编码器 → AirPlay/网络接收端
```

潜在路线图：

- 先做本地设备路由
- 第二步做 RTP/RAOP/AirPlay 研究验证
- 通过 mDNS/Bonjour 发现接收端
- 后续做延迟补偿和同步

除非 AirPlay 是首要需求，否则网络流播不应是第一里程碑，因为虚拟设备和路由基础仍是必须的。

## MVP 定义

MVP 应验证产品能实现 DAW 级音频路由的独立替代方案。

MVP 能力：

- 先在 Windows 运行，后 macOS
- 登录时作为后台引擎启动
- 列出物理和虚拟音频设备
- 创建简单的立体声路由
- 保存和恢复一个配置
- 处理设备断开/重连
- 提供一个简陋的 GUI 路由矩阵
- 使用现有虚拟音频设备而非自研驱动

MVP 非目标：

- 自研虚拟驱动
- AirPlay 流播
- 应用级路由
- 环绕声路由
- 高级调音台效果
- 跨多设备采样级精确同步

## 关键技术风险

- 虚拟设备实现是驱动级别的工作，不是普通应用代码
- Windows 驱动签名和 macOS 公证可能主导项目排期
- 多个硬件设备之间的路由需要时钟漂移校正
- 低延迟和无爆音运行需要原生实时代码
- 应用级路由在不同平台差异很大
- AirPlay 兼容性可能牵涉协议和许可证敏感性

## 第一个具体里程碑

构建一个桌面原型，能够：

1. 列出可用的音频输入/输出设备
2. 让用户选择一个输入和一个输出
3. 持续将立体声从输入传递到输出
4. 将该路由保存到配置文件
5. 重新启动后自动恢复该路由

这可以在投入定制虚拟驱动之前验证核心路由层。
