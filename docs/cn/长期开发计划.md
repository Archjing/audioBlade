# audioBlade 长期开发计划

## 计划定位

本文档基于 `docs/TECHNICAL_PLAN.md` 和 `docs/DEVELOPMENT_GUIDELINES.md` 制定，用于把 audioBlade 从策划项目推进为可持续开发的跨平台桌面音频路由产品。

audioBlade 的长期目标不是做一个依赖 DAW 的音频工作流插件，而是做一个可独立运行的系统级音频路由应用：

- Windows + macOS 桌面应用
- 后台音频引擎独立于 GUI 存活
- 虚拟音频接口与物理音频设备之间可路由
- 路由配置可保存、可恢复、可在设备重连后重新建立
- 未来可扩展到网络输出、AirPlay 类流播和更复杂的多通道路由

## 长期原则

### 先做产品模型，再做自研驱动

自研虚拟音频驱动是长期能力，但不应作为第一阶段目标。第一阶段应使用许可证兼容的开源虚拟设备验证音频路由、配置持久化、后台生命周期和设备重连。

只有当第三方开源驱动在商业打包、稳定性、延迟、通道数、品牌控制或安装体验上成为明确障碍时，才启动第一方驱动研发。

### 实时音频路径保持原生、简单、可预测

实时音频路径由 C++/JUCE 承担。Rust、GUI、IPC 和脚本工具只处理控制、配置、诊断和生命周期，不在常规运行路径中搬运实时音频缓冲区。

音频回调中不得执行不可预测操作，包括动态分配、无界锁、阻塞 I/O 和复杂日志写入。

### GUI 不是引擎

GUI 负责编辑路由、配置、状态和诊断展示。后台引擎负责真实音频路由。长期目标是 GUI 崩溃或退出时，已经建立的路由不应立即中断。

MVP 可以先用 JUCE 单进程实现，但代码边界必须区分音频平面和控制平面，为后续拆分 Rust 控制服务和后台引擎预留空间。

### Windows 先行，macOS 跟进

Windows 是第一验证平台，因为现有方案中 `VirtualDrivers/Virtual-Audio-Driver` 更贴近初始虚拟扬声器 + 虚拟麦克风用例。

macOS 不应被忽略，但应在 Windows 路由产品模型跑通后再验证 Roc VAD、Core Audio/HAL/AudioDriverKit 相关路径。

## 阶段 0：项目骨架与技术基线

目标：把策划项目变成可构建、可运行、可测量的工程项目。

主要交付：

- CMake + JUCE 项目骨架
- 最小桌面应用入口
- 音频设备枚举页面或命令行诊断工具
- 路由配置 schema 初稿
- 日志、崩溃信息和运行环境采集基础设施
- `docs/adr/` 技术决策记录目录

验收标准：

- Windows 上可以构建并启动应用
- 应用可以列出输入、输出设备及其采样率、通道数、默认状态
- 项目可以记录一次设备枚举结果作为诊断输出
- 没有把 Python、GUI 或 IPC 放进实时音频路径

建议周期：1-2 周。

## 阶段 1：Windows 本地立体声路由原型

目标：证明 audioBlade 可以在没有 DAW 的情况下持续转发音频。

主要交付：

- Windows WASAPI 设备枚举
- 一路输入到一路输出的立体声转发
- 基础缓冲区处理
- 简单启动/停止控制
- 初版延迟、CPU、掉音统计

验收标准：

- 可选择一个输入设备和一个输出设备
- 可连续转发立体声音频至少 30 分钟
- 记录采样率、缓冲区大小、CPU 占用和掉音情况
- 关闭 GUI 后的引擎存活策略有明确结论：先支持或明确暂缓

建议周期：2-4 周。

关键风险：

- 不同设备采样率不一致
- 输入输出设备来自不同硬件时存在时钟漂移
- Windows 共享模式和独占模式行为不同

## 阶段 2：虚拟音频端点 MVP

目标：让 audioBlade 具备虚拟声卡/虚拟音频接口参与路由的能力。

Windows 起始依赖：

- `VirtualDrivers/Virtual-Audio-Driver`

主要交付：

- 安装和检测虚拟扬声器、虚拟麦克风
- 虚拟输入到物理输出的 stereo route
- 物理输入到虚拟输出的 stereo route
- 设备缺失时的可读错误提示
- 许可证、安装、卸载和商业使用风险记录

验收标准：

- Windows 应用可以识别虚拟端点
- 用户可以把系统或应用播放送入 audioBlade 虚拟输入，再输出到物理声卡
- 用户可以把物理麦克风/声卡输入送到虚拟麦克风，供会议软件或录音软件选择
- 重启应用后可以恢复上一条路由

建议周期：3-6 周。

退出条件：

- 如果虚拟驱动无法稳定安装或正常暴露端点，必须记录阻塞原因，并决定改用 SYSVAD 研究路径、其他许可证兼容驱动，或缩小 MVP。

## 阶段 3：持久化路由与后台生命周期

目标：接近 TuneBlade 类“开机后持续可用”的产品体验。

主要交付：

- Profile 持久化
- 登录后自动启动
- 后台引擎生命周期管理
- GUI 连接/断开后台引擎
- 设备重连恢复策略
- 稳定设备 ID 与人类可读 fallback 匹配

Profile 最小模型：

```json
{
  "name": "Default Monitor Route",
  "routes": [
    {
      "source": "virtual:AudioBlade Input:1-2",
      "destination": "device:External Interface Out:1-2",
      "gainDb": 0.0,
      "mute": false
    }
  ],
  "sampleRatePolicy": "preferDestination"
}
```

验收标准：

- 重启应用后恢复路由
- 系统重新登录后恢复路由
- 拔插 USB 声卡后能自动恢复或给出明确修复动作
- GUI 退出不等同于路由立即失效，或至少有清晰产品说明

建议周期：4-8 周。

关键风险：

- Windows 设备 ID 在重启或重连后变化
- 自动启动权限和用户安全软件拦截
- 引擎和 GUI 的状态一致性

## 阶段 4：基础路由矩阵与可用产品 Alpha

目标：从技术原型变成用户能理解、能反复使用的桌面工具。

主要交付：

- 路由矩阵 UI
- 每路 gain/mute
- Profile 新建、复制、删除、切换
- 设备状态面板
- 采样率/缓冲区/延迟诊断
- 基础问题修复指引

验收标准：

- 非开发者用户可以完成常见路由：系统播放到外置声卡、物理麦克风到虚拟麦克风
- 应用能清楚显示当前路由是否正在工作
- 常见错误能被解释：设备不存在、采样率冲突、端点被独占、权限不足
- 至少 5 个真实应用组合通过测试：浏览器、会议软件、播放器、录音软件、一个 DAW

建议周期：6-10 周。

## 阶段 5：macOS 验证与跨平台抽象

目标：证明 audioBlade 架构不是 Windows-only，并建立 macOS 可行路径。

macOS 起始依赖：

- `roc-streaming/roc-vad`

主要交付：

- macOS 设备枚举
- Roc VAD 安装、检测、卸载验证
- 虚拟端点到物理设备的 stereo route
- macOS 权限、安装、公证风险记录
- 与 Windows 路由模型共用的 Profile schema

验收标准：

- macOS 上普通应用可以看到虚拟音频端点
- audioBlade 可以在虚拟端点和物理设备之间转发音频
- 重启、重新登录、设备重连场景有明确测试记录
- Roc VAD 的 MPL-2.0 义务、vendoring/forking 要求和商业打包风险已记录

建议周期：6-12 周。

决策门槛：

- 如果 Roc VAD 不适合作为本地环回基础，则评估 Soundflower 兼容性或进入第一方 Core Audio/HAL 原型研究。
- BlackHole 只能作为功能基准，不能作为专有商业版默认依赖，除非授权路径改变。

## 阶段 6：生产级音频稳定性

目标：提升到可长时间运行、可诊断、可支持的产品质量。

主要交付：

- Sample-rate conversion
- 多设备时钟漂移检测和校正
- WASAPI exclusive mode 选项
- ASIO 支持研究与原型
- 多通道路由基础
- 更完整的日志和诊断包导出
- 长时间稳定性测试

验收标准：

- 常见 44.1 kHz / 48 kHz 混用场景可用
- 外置声卡和虚拟端点长时间运行无明显掉音
- 设备断开、睡眠唤醒、默认设备变化后有可预测恢复行为
- 24 小时稳定性测试有记录

建议周期：2-4 个月。

## 阶段 7：第一方虚拟驱动研发决策

目标：决定是否把虚拟音频设备从第三方依赖转为自研资产。

启动条件至少满足一项：

- 第三方驱动无法满足商业打包或品牌要求
- 安装体验不可控
- 稳定性或延迟无法达到目标
- 通道数或端点拓扑不足
- 产品已有足够需求支撑数月驱动投入

Windows 路线：

- 基于 Microsoft SYSVAD 样例研究
- 设计 audioBlade 虚拟 speaker/mic 端点
- 建立驱动签名和安装流程
- 测试 WASAPI、DirectSound、会议软件、浏览器、DAW

macOS 路线：

- 评估 AudioDriverKit 与 Core Audio HAL/AudioServerPlugIn 路径
- 实现虚拟输入/输出流
- 处理 bundle identifier、entitlement、公证和用户批准
- 测试 Apple Silicon 和目标 macOS 版本

建议周期：

- Windows 可用第一方驱动：3-6 个月
- macOS 可用第一方驱动：4-8 个月
- 双平台商业级驱动：9-18 个月，视团队规模和测试深度而定

## 阶段 8：网络输出与 TuneBlade 类能力

目标：在本地路由稳定后，将网络接收端作为新的 sink 类型引入。

主要交付：

- 网络 sink 抽象
- mDNS/Bonjour 发现研究
- AirPlay/RAOP 类协议研究
- 编码、缓冲、延迟补偿
- 多接收端同步策略

验收标准：

- 网络输出作为路由图中的一种目标节点存在
- 本地路由不因网络 sink 引入而复杂化
- 网络延迟和同步问题有独立诊断

建议周期：2-6 个月。

注意：网络输出不应早于本地虚拟端点路由能力。TuneBlade 类能力是长期扩展，不是第一 MVP 的核心。

## 产品里程碑

### Prototype

内部技术验证版。

必须证明：

- 可枚举设备
- 可转发立体声音频
- 可保存和恢复一条路由

### MVP

面向早期用户的最小可用版。

必须证明：

- Windows 上虚拟端点参与路由
- Profile 持久化
- 后台运行
- 基础 UI 可操作
- 设备重连有处理策略

### Alpha

真实工作流试用版。

必须证明：

- 常见应用组合可用
- 错误可诊断
- 长时间运行基本稳定
- macOS 路径已有可行性结论

### Beta

准商业测试版。

必须证明：

- 安装、更新、卸载可控
- 崩溃和诊断可收集
- 采样率不匹配、设备变化、睡眠唤醒可处理
- 许可证和驱动依赖风险有正式记录

### Commercial Release

商业发布版。

必须证明：

- 双平台或明确的首发平台策略
- 驱动依赖法律和工程风险已审查
- 支持流程和诊断包完善
- 有明确的第一方驱动或第三方驱动长期策略

## 不建议现在做的事

- 不要一开始做 AirPlay 或多接收端网络同步
- 不要一开始自研 Windows 和 macOS 双平台驱动
- 不要把 Python 放进实时音频路径
- 不要把 GUI 设计成路由存活的必要条件
- 不要把 VoiceMeeter、VB-CABLE、BlackHole 当作默认商业依赖
- 不要承诺完整替代 VoiceMeeter Banana/Potato，除非后续明确投入完整虚拟调音台能力

## 最近三个月建议路线

第 1 个月：

- 建立 JUCE + CMake 项目骨架
- 完成 Windows 设备枚举
- 完成输入到输出的立体声转发
- 定义 Profile schema 初稿

第 2 个月：

- 接入 Windows 虚拟音频端点
- 完成虚拟输入到物理输出
- 完成物理输入到虚拟输出
- 实现路由保存和重启恢复

第 3 个月：

- 建立后台生命周期模型
- 完成设备重连处理第一版
- 做基础路由矩阵 UI
- 形成 Alpha 前的技术风险清单

三个月后的关键决策：

- Windows 第三方虚拟驱动是否足够继续支撑 MVP
- 是否开始 macOS Roc VAD 验证
- 是否需要提前启动第一方 Windows 驱动研究
- GUI 是否继续用 JUCE，还是在引擎稳定后引入 Tauri

## 项目管理规则

- 每个阶段必须有可运行构建或可复现实验结果
- 每个驱动决策必须记录许可证、商业使用评估、功能差距和打包风险
- 每个音频稳定性问题必须附带复现环境、采样率、缓冲区、设备名和应用名
- 每个主要架构变化必须记录 ADR
- 每个发布里程碑必须包含安装、卸载、恢复和诊断验证
