# audioBlade 开发规范

## 目的

本文档定义 audioBlade 的项目级工程规则。它将当前的技术推理转化为实际的开发约束。

详细推理和参考来源见 `docs/TECHNICAL_PLAN.md`（英文版，此文件为中文对照 `docs/技术方案.md`）。

## 产品边界

audioBlade 是一款 Windows 和 macOS 的持久化桌面音频路由器。

产品必须提供：

- 无需 DAW 即可保持的系统级音频路由
- 虚拟音频端点与物理音频设备之间的路由
- 可保存的配置，重启或设备重连后恢复路由
- 独立于 GUI 的后台音频引擎
- 用于编辑路由、增益、静音状态和配置的 UI

产品不得依赖于任何 DAW、编辑器或仅前台运行的进程来保持路由存活。

## 架构规则

使用以下架构拆分：

- `engine`：实时音频路由、图状态、设备桥接、配置恢复
- `control`：Rust 控制服务、路由图所有权、配置、设备策略、诊断
- `app/gui`：路由编辑器和状态 UI
- `platform/windows`：WASAPI、ASIO、驱动集成、Windows 安装包细节
- `platform/macos`：Core Audio、AudioDriverKit/HAL 集成、macOS 打包细节
- `drivers`：第三方驱动集成说明或自研驱动实验
- `docs`：技术决策、参考来源和风险说明
- `tools`：诊断、脚本和测试工具

保持音频引擎与 GUI 分离。如果后台引擎正在运行，GUI 崩溃或退出不应导致现有路由中断。

对于第一个 MVP，控制服务可以在 JUCE 应用内实现以减少活动部件。代码中仍应保持控制平面和音频平面的职责分离，以便后续可以拆分出 Rust 服务。

## 语言和运行时规则

默认长期技术栈：

- 实时音频引擎工作使用 C++
- 跨平台音频设备管理和首个 MVP GUI 使用 JUCE
- 初始音频路径验证通过后，控制服务使用 Rust
- 仅当更丰富的 Web 风格 GUI 变得比最小的原生 JUCE 应用路径更有价值时，才使用 Tauri
- 构建编排使用 CMake
- Python 仅用于工具、诊断、脚本和测试

不得将实时音频回调代码放在 Python 中。

不得依赖 Python 向操作系统暴露虚拟音频设备。

不得将正常的实时音频缓冲区通过 Rust、IPC 或 GUI 传递。Rust 可以控制路由和状态，但 C++ 拥有实时音频路径。

MVP 技术栈规则：

- 如果 C++/JUCE 单进程实现能更快验证音频路径，则先使用它
- 当配置管理、诊断、更新处理和设备策略的复杂度足以证明拆分边界的合理性时，再拆分 Rust 控制服务

## 驱动依赖规则

所有默认虚拟音频驱动依赖必须开源并兼容未来商业使用。

首选许可证：

- MIT
- BSD
- Apache-2.0
- MS-PL
- MPL-2.0

仅在明确审核后允许：

- LGPL
- 文件级 Copyleft（义务可隔离且可管理）

在专有商业构建中不得作为默认依赖使用：

- GPL-2.0 驱动
- GPL-3.0 驱动
- 闭源免费软件
- 捐赠软件驱动
- 专有虚拟线缆产品

在商业安装包中捆绑任何驱动前，须进行正式许可证审查。本文档是工程策略，非法律建议。

## 起始驱动决策

### Windows

从 `VirtualDrivers/Virtual-Audio-Driver` 开始。

理由：

- 已支持 Windows 10/11
- 暴露虚拟扬声器和虚拟麦克风
- 许可证路径更兼容未来商业使用：原创代码 MIT，SYSVAD 衍生部分为 MS-PL
- 与未来 SYSVAD 派生的 Windows 自研驱动方向一致
- 比从原始 Microsoft SYSVAD 起步更快完成 MVP 验证

除非任务是明确的 Windows 自研驱动研发，否则不要从原始 SYSVAD 开始。

将 VoiceMeeter 视为 Windows 功能基准，而非依赖项。

不得将 VB-CABLE 或 VoiceMeeter 作为默认依赖，因为它们不是开源商业依赖。

### macOS

以 `roc-streaming/roc-vad` 作为首个 macOS 验证候选启动。

Roc VAD 验证必须回答以下问题：

- 它能在目标 macOS 版本上创建虚拟音频端点吗？
- 普通应用和 DAW 能看到这些端点吗？
- audioBlade 能否在 Roc VAD 端点和物理设备之间路由音频？
- 延迟对于第一个 MVP 可接受吗？
- 设备能经受重启、重新启动和重连场景吗？
- 对用户来说安装可接受吗？
- MPL-2.0 义务对于预期的商业模型是否可管理？

Soundflower 仅作为兼容性参考，尽管其 MIT 许可证有吸引力，但该项目较旧。

BlackHole 作为 macOS 功能基准，不作为默认依赖。除非 audioBlade 兼容 GPL-3.0 或从作者处获得单独商业许可，不得将 BlackHole 捆绑到专有商业构建中。

## MVP 顺序

按此顺序构建：

1. Windows 设备枚举
2. 使用 VirtualDrivers/Virtual-Audio-Driver 的 Windows 路由引擎
3. 一路从虚拟输入到物理输出的立体声路由
4. 一路从物理输入到虚拟输出的立体声路由
5. 配置保存和恢复
6. 后台引擎生命周期
7. 设备断开和重连恢复
8. 基础路由矩阵 UI
9. macOS Roc VAD 验证
10. macOS 路由引擎集成

不要从 AirPlay、网络流播、多接收端同步或自研驱动开始。

## 自研驱动策略

自研虚拟音频驱动是可行的，但不得作为第一里程碑。

仅在满足以下至少一个条件时再启动自研驱动工作：

- 第三方开源驱动阻止了商业打包
- 端点可靠性不可接受
- 品牌或安装包要求需要自有驱动
- 通道数、延迟或路由行为无法满足产品需求
- 应用已验证了足够的需求，值得投入数月驱动工作

预期工作量：

- Windows 可用自研驱动：额外 3-6 个月
- macOS 可用自研驱动：额外 4-8 个月
- 双平台商业就绪：实际视团队规模和测试深度，约 9-18 个月

将 Windows 和 macOS 驱动视为独立项目。不要假设有一个跨平台驱动层存在。

## 实时音频规则

实时音频代码必须：

- 避免在音频回调中分配内存
- 避免在音频回调中加锁（除非已证明是有界且安全的）
- 避免在音频回调中执行阻塞 I/O
- 保持回调工作可预测且小巧
- 将控制平面变更与音频线程处理分离
- 显式处理采样率和缓冲区大小变更
- 在实时路径之外记录诊断信息

路由必须考虑：

- 采样率不匹配
- 缓冲区大小不匹配
- 多设备时钟漂移
- 设备断开
- 重启或重连后设备 ID 变化
- Windows 上独占/共享模式的差异

## 路由模型规则

使用图模型，而非硬编码设备对。

必需的概念：

- `AudioNode`：物理设备、虚拟设备、应用捕获流、网络接收端
- `Port`：节点上的输入或输出通道
- `Route`：从源端口到目标端口的连接
- `Profile`：命名的路由集，含增益、静音、通道映射和采样率策略

所有路由必须可序列化。

配置应优先使用稳定的设备标识符，并在 ID 变化时配合易读的备选匹配机制。

## 范围控制

MVP 包含：

- 设备列表
- 简单的立体声路由
- 路由持久化
- 后台引擎
- 基础 GUI
- 设备重连处理
- 许可证兼容的虚拟驱动依赖

MVP 排除：

- AirPlay 流播
- 应用级路由
- 自研虚拟驱动
- 环绕声路由
- 效果处理
- 完整的 VoiceMeeter 替代功能
- 多接收端网络同步

## 基准产品

仅将这些产品用作行为参考：

- Windows：VoiceMeeter
- Windows：VB-CABLE
- macOS：BlackHole
- macOS：Soundflower（历史兼容性）
- TuneBlade（网络/AirPlay 类行为参考）

不得复制专有实现细节。

不得将非开源产品作为项目依赖。

## 文档规则

每项重大技术决策必须记录在 `docs/TECHNICAL_PLAN.md` 或专用的 ADR 风格文档中。

与驱动相关的决策必须包含：

- 仓库 URL
- 许可证 URL
- 商业使用评估
- 功能差距评估
- 打包风险
- 决策状态：候选、已采纳、已拒绝、或仅作基准

项目决策应使用清晰的语言。避免使用歧义术语，除非文档中明确定义了它们。
